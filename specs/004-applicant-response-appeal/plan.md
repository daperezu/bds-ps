# Implementation Plan: Applicant Response & Appeal

**Branch**: `004-applicant-response-appeal` | **Date**: 2026-04-17 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/004-applicant-response-appeal/spec.md`

## Summary

Add the applicant-response stage to the application lifecycle. After reviewers finalize their decisions (spec 002), the applicant is shown each item with its reviewer decision and must accept or reject each one individually. If at least one item is applicant-rejected, the applicant may open a single appeal ‚Äî a text-only dispute thread with any reviewer. Appeals freeze the application until resolved as **uphold** (decision stands; application advances), **grant‚Äîreopen to draft** (applicant re-edits), or **grant‚Äîreopen to review** (reviewers re-decide). Appeal count is capped by a new `SystemConfiguration` key. Introduces three new domain aggregates (`ApplicantResponse` with child `ItemResponse`, `Appeal` with child `AppealMessage`), extends the `ApplicationState` enum with two new states (`AppealOpen`, `ResponseFinalized`), and adds a new MVC controller for the applicant response UI.

## Technical Context

**Language/Version**: C# / .NET 10.0
**Primary Dependencies**: ASP.NET MVC, Entity Framework Core 10.0, ASP.NET Identity, .NET Aspire
**Storage**: SQL Server (Aspire-managed container for dev, dacpac schema management)
**Testing**: Playwright for .NET (NUnit), NUnit (unit tests), NSubstitute (mocking)
**Target Platform**: Linux/Windows server (Aspire-orchestrated)
**Project Type**: Web application (server-side MVC, no SPA)
**Performance Goals**: Response UI renders in <500ms for applications with up to 10 items √ó 5 quotations; dispute thread renders in <500ms for threads with up to 100 messages
**Constraints**: All state transitions must be gated by domain methods (no external state manipulation); optimistic concurrency for concurrent reviewer replies on the same appeal
**Scale/Scope**: Typical application has 1-10 items; typical dispute thread has 2-20 messages; appeal cap is expected to be 1-3 in production configuration

## Complexity Tracking

No violations. All changes align with constitution principles.

| Decision | Justification |
|----------|---------------|
| Persist `ApplicantResponse` as snapshot (vs. reconstruct from item state) | Audit requirement (FR-024) and immutability requirement (FR-006) both simplified by explicit snapshot. Reconstruction would need a separate audit log entity anyway. |
| `AppealMessage` is an entity (not a value object) | Needs a stable identity for author attribution, future querying (who posted what when), and safe addition via DbContext. Value-object collections in EF Core require owned-entity configuration that's no simpler than a child entity. |
| Two new `ApplicationState` values (AppealOpen, ResponseFinalized) | Explicit state is more debuggable and testable than deriving state from "existence of active Appeal" flags. The enum values are additive (preserve existing 0-3), so no data migration. |
| New controller (`ApplicantResponseController`) rather than extending `ApplicationController` | Separation of responsibility: the applicant's response flow is distinct from the application authoring/draft flow. Keeps controllers focused. |
