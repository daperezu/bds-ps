# Data Model: Funding Agreement Document Generation

**Feature**: 005-funding-agreement-generation
**Date**: 2026-04-17

This document defines the domain data model for the Funding Agreement generation feature. It introduces one new aggregate root (`FundingAgreement`) and adds one navigation + one domain method group on the existing `Application` aggregate. No existing entities require structural changes.

---

## 1. New Aggregate: `FundingAgreement`

### Purpose

Records the system-generated PDF artifact produced for an `Application`. Owns the file metadata, the generating-user audit, and the concurrency token that serializes parallel generation attempts.

### Fields

| Field | Type | Nullable | Purpose / Rule |
|-------|------|----------|---------------|
| `Id` | `int` | no | Primary key. EF Core identity column. |
| `ApplicationId` | `int` | no | Foreign key to `Applications.Id`. Unique (one agreement per application). |
| `FileName` | `string` (max 260) | no | Opaque storage filename. Generated by `IFileStorageService.SaveFileAsync`. |
| `ContentType` | `string` (max 100) | no | Always `"application/pdf"` in this feature; kept as a field for parity with `Document`. |
| `Size` | `long` | no | File size in bytes. |
| `StoragePath` | `string` (max 500) | no | Absolute or root-relative storage path returned by `IFileStorageService`. Never exposed to the UI. |
| `GeneratedAtUtc` | `DateTime` | no | Timestamp of the current generation (set on create and replaced on regeneration). |
| `GeneratedByUserId` | `string` (max 450) | no | ASP.NET Identity user id of the acting administrator or reviewer. |
| `RowVersion` | `byte[]` (`rowversion`) | no | EF Core concurrency token. Mirrors the pattern on `Application`. |

No `Id` is exposed to the client; the download URL is keyed on `ApplicationId`.

### Relationships

- **`Application` 1..1 (owns) → `FundingAgreement` 0..1**. The FK lives on `FundingAgreement.ApplicationId`. A unique index on `ApplicationId` enforces 0..1.
- **No cascade delete**. `Application` is not deleted in this product's lifecycle; the agreement must be orphaned explicitly if ever needed.

### Invariants

- **Inv-A**: `FundingAgreement` may only be constructed through `Application.GenerateFundingAgreement(...)` or `Application.RegenerateFundingAgreement(...)`. The constructor is internal to the Domain project.
- **Inv-B**: `ContentType` MUST be `"application/pdf"`.
- **Inv-C**: `Size` MUST be `> 0` — a zero-length PDF is not a valid agreement.
- **Inv-D**: `StoragePath`, `FileName`, and `GeneratedByUserId` MUST be non-empty.

### State

`FundingAgreement` has a single logical state ("current"). Regeneration mutates fields in place (with a new `RowVersion`) rather than creating a new row. There is no version history.

## 2. `Application` extensions

The existing `Application` aggregate (spec 001, modified through specs 002 and 004) gains one navigation and a small set of methods. No existing members change.

### New navigation

```csharp
public FundingAgreement? FundingAgreement { get; private set; }
```

### New domain methods

#### `CanGenerateFundingAgreement(out IReadOnlyList<string> errors) : bool`

Evaluates all four FR-002 preconditions server-side and returns:
- `true` with an empty `errors` list when all preconditions hold, or
- `false` with a list of user-presentable error messages (one per failed precondition).

Precondition logic:
- `State == ApplicationState.ResponseFinalized` (not `Resolved`, not `AppealOpen`, not earlier states).
- No `Appeal` with `Status == AppealStatus.Open` in `Appeals`.
- The most recent `ApplicantResponse` exists and has at least one `ItemResponse` with `Decision == ItemResponseDecision.Accept`.
- Every `ItemResponse` in that response has `Decision != null` (guaranteed by the response submission logic already in the codebase, but asserted defensively).

#### `GenerateFundingAgreement(string fileName, string contentType, long size, string storagePath, string generatingUserId) : FundingAgreement`

- Preconditions: `CanGenerateFundingAgreement(out _)` is `true`; `FundingAgreement is null`.
- Throws `InvalidOperationException` when preconditions fail, wrapping the failed-precondition messages.
- Constructs a new `FundingAgreement` and assigns it to `FundingAgreement`.
- Returns the new entity so the caller can use its metadata.

#### `RegenerateFundingAgreement(string fileName, string contentType, long size, string storagePath, string regeneratingUserId) : FundingAgreement`

- Preconditions: `CanGenerateFundingAgreement(out _)` is `true`; `FundingAgreement is not null`.
- Mutates the existing `FundingAgreement`'s metadata in place:
  - `FileName`, `ContentType`, `Size`, `StoragePath` replaced.
  - `GeneratedAtUtc` set to `DateTime.UtcNow`.
  - `GeneratedByUserId` set to `regeneratingUserId`.
- EF Core's concurrency token check on `RowVersion` serializes parallel callers.
- Returns the mutated `FundingAgreement`.

#### `CanUserAccessFundingAgreement(User user, IUserRoleProvider roles) : bool`

Returns `true` iff any of:
- `user` is `Applicant` AND `user.Id == this.Applicant.UserId`.
- `user` is `Administrator`.
- `user` is `Reviewer` AND `user` was assigned to this application's review (existing join already tracked in spec 002).

Otherwise `false`. Non-disclosing responses at the controller layer make "no agreement exists" and "not authorized" return the same HTTP response shape.

#### `CanUserGenerateFundingAgreement(User user, IUserRoleProvider roles) : bool`

Same as the access check, except the `Applicant` branch returns `false`. The check is combined at the controller with `CanGenerateFundingAgreement(...)` to decide whether the "Generate" / "Regenerate" actions are available.

## 3. Validation Rules

| Rule | Source FR | Enforcement Point |
|------|-----------|-------------------|
| Generation requires all four preconditions. | FR-002 | `Application.CanGenerateFundingAgreement` |
| Applicants cannot generate or regenerate. | FR-003 | `Application.CanUserGenerateFundingAgreement` |
| Stale-UI requests rejected server-side. | FR-004 | Controller re-checks `CanGenerateFundingAgreement` at action time. |
| At most one current agreement per application. | FR-011 | Unique index on `FundingAgreements.ApplicationId` + `Application.GenerateFundingAgreement` guard. |
| Regeneration overwrites prior file and metadata. | FR-013, FR-014 | `Application.RegenerateFundingAgreement` + compensating delete of prior storage path in the use-case orchestrator. |
| Regeneration unavailable when preconditions fail after prior generation. | FR-015 | UI reads `CanGenerateFundingAgreement`; server re-checks. |
| Non-disclosure of agreement existence. | FR-019 | Controller returns 404 for both "no agreement" and "not authorized" cases. |
| Concurrency serialization. | FR-022 | `RowVersion` on `FundingAgreement`; `DbUpdateConcurrencyException` → retriable error. |

## 4. Persistence / Schema

### Table: `FundingAgreements`

| Column | Type | Nullable | Notes |
|--------|------|----------|-------|
| `Id` | `INT IDENTITY(1,1)` | NOT NULL | PK |
| `ApplicationId` | `INT` | NOT NULL | FK → `Applications.Id`; unique |
| `FileName` | `NVARCHAR(260)` | NOT NULL | |
| `ContentType` | `NVARCHAR(100)` | NOT NULL | |
| `Size` | `BIGINT` | NOT NULL | `> 0` check |
| `StoragePath` | `NVARCHAR(500)` | NOT NULL | |
| `GeneratedAtUtc` | `DATETIME2(3)` | NOT NULL | |
| `GeneratedByUserId` | `NVARCHAR(450)` | NOT NULL | FK → `AspNetUsers.Id` |
| `RowVersion` | `ROWVERSION` | NOT NULL | Auto-managed |

### Indexes & constraints

- **PK**: `Id`.
- **UQ**: `ApplicationId` (enforces 0..1 per application).
- **FK**: `ApplicationId → Applications(Id)`, `NO ACTION` on delete.
- **FK**: `GeneratedByUserId → AspNetUsers(Id)`, `NO ACTION` on delete.
- **CHECK**: `Size > 0`.

### EF Core configuration highlights

- Map the aggregate as owned-by-`Application` via a configured relationship (private backing field `_fundingAgreement` on `Application`; EF-backing-field binding).
- `RowVersion` as concurrency token.
- `Application`'s DbContext `OnModelCreating` registers the new entity configuration (`FundingAgreementConfiguration`).

## 5. Touch Points on Existing Entities

- `Application`: add `FundingAgreement` navigation; add domain methods listed in section 2.
- `ApplicantResponse` / `ItemResponse` / `Appeal`: **read-only** from the perspective of this feature. The precondition check reads the latest response and the open-appeal collection; no mutation.
- `Item` / `Quotation` / `Supplier`: **read-only**. The PDF template reads item description, category, accepted quotation's supplier, and price.
- `Applicant`: **read-only**. Applicant identity fields are read at generation time.
- `SystemConfiguration`: **no changes**. Locale and Funder options are read from `appsettings`, not `SystemConfiguration` (R-004, R-006).

## 6. Seed Data

No seed data is required for this feature. The `FundingAgreements` table starts empty and is populated only by the use case.

The Razor template partial (`_FundingAgreementTermsAndConditions.cshtml`) ships with a visibly-marked placeholder paragraph until legal delivers the canonical copy (R-005).

---

## 7. Out-of-model Notes

- Prior generated files are deleted from storage on regeneration (compensating action in the use case orchestrator). No "soft delete" row remains in `FundingAgreements`.
- If the container's file storage is ever migrated (e.g., local disk → S3), the `IFileStorageService` implementation changes; nothing in this data model does.
- Post-signature regeneration lockout (future Digital Signatures spec) will add a new predicate to `CanGenerateFundingAgreement` without changing any field on this aggregate.
