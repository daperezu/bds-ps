# Review Brief: Funding Agreement Document Generation

**Spec:** specs/005-funding-agreement-generation/spec.md
**Generated:** 2026-04-17

> Reviewer's guide to scope and key decisions. See full spec for details.

---

## Feature Overview

After applicants respond to a reviewed application (spec 004), the platform needs a concrete, downloadable artifact that captures the funded outcome: the formal Funding Agreement. This feature produces that document as a single PDF per application, generated by an administrator or reviewer once every approved item has a final response, no appeal is active, and at least one item is accepted. Applicants, administrators, and reviewers can download the agreement from the application page; administrators and reviewers can regenerate it (overwriting the prior file) while preconditions still hold. The generated PDF is the canonical bytes that Digital Signatures, Payment, and Closure (all future specs) will build on.

## Scope Boundaries

- **In scope:** Admin/reviewer-triggered PDF generation of the Funding Agreement; preconditions gating the action; content model (funder, applicant, accepted items table, total, hardcoded T&Cs, empty signature blocks); persistence through the existing file storage abstraction; regeneration with confirmation that overwrites the prior file; authenticated download for applicant/admin/reviewer; non-disclosing authorization; optimistic-concurrency serialization of parallel attempts; structured logging.
- **Out of scope:** Any other document type (submission receipts, decision letters, appeal resolutions, closure letters); digital signing; notifications; payment orders / application closure; admin-editable or versioned templates; multi-language content; amendments/addenda; bulk export; PDF accessibility beyond clean rendering; formal retention policy.
- **Why these boundaries:** The Funding Agreement is the *only* document that unblocks the next workflow stages. Adding more document types, an editing UI, or versioning before we know they're needed is YAGNI. The template stays hardcoded in code; a future feature can switch to admin-editable templates if more document types arrive.

## Critical Decisions

### Scope limited to the Funding Agreement

- **Choice:** Ship one document type. Reject the "full lifecycle set" and "admin-configurable templates" alternatives.
- **Trade-off:** Smallest deliverable that unblocks signatures/payment vs. upfront flexibility for future documents.
- **Feedback:** If the roadmap is likely to demand 3+ document types soon, is it worth paying the admin-configurable-template cost now instead of refactoring later?

### Admin/reviewer action after response is fully resolved

- **Choice:** A human clicks "Generate agreement". No auto-trigger on response resolution.
- **Trade-off:** Explicit checkpoint before a near-legal document is materialized vs. hands-off automation.
- **Feedback:** Is the manual step the intended control, or would stakeholders prefer a one-click/auto path once response is complete?

### Hardcoded template (Razor + Syncfusion HTML-to-PDF)

- **Choice:** Layout and wording live in the Web project as a Razor view; Syncfusion's HTML-to-PDF converter is the rendering library.
- **Trade-off:** Wording changes require a deploy; gained: no browser runtime dependency and familiar Razor ergonomics for developers. See `implementation-notes.md` for the Syncfusion vs. QuestPDF vs. Playwright analysis.
- **Feedback:** Is the commercial Syncfusion license acquirable on the expected timeline, and does the stakeholder accept deploy-bound wording changes?

### Regenerate with confirmation; only the latest retained

- **Choice:** No version history. Regeneration overwrites the prior PDF, with an explicit confirmation dialog. Admin *and* reviewer can regenerate.
- **Trade-off:** Simple storage and UI vs. no audit trail of prior agreement contents. The future Digital Signatures feature will introduce the lockout-once-signed rule, so overwrite is only viable before signing.
- **Feedback:** Is a prior-version audit trail a stakeholder expectation? If yes, better to introduce versioning now than retrofit after signatures ship.

### Block generation when zero accepted items

- **Choice:** The "Generate agreement" action is disabled when the applicant rejected everything. Terminal closure of all-rejected applications is deferred to Payment & Closure.
- **Trade-off:** No "no-fund acknowledgment" document at this stage; cleaner single-purpose Funding Agreement.
- **Feedback:** Is the absence of a paper trail for all-rejected applications acceptable until the Closure feature lands?

## Areas of Potential Disagreement

### Reviewer regeneration rights (not just admin)

- **Decision:** Both administrators and reviewers can regenerate the agreement.
- **Why this might be controversial:** Regeneration is a destructive, audit-visible action touching a near-legal document. Many organizations reserve that kind of write to an elevated role.
- **Alternative view:** Restrict regeneration to administrators only; reviewers retain read/download access.
- **Seeking input on:** Whether the operational convenience of reviewer-regeneration is worth the widened blast radius, given no separation-of-duties gate exists yet.

### Overwrite-on-regeneration with no version history

- **Decision:** Prior PDFs are deleted when a new one is generated.
- **Why this might be controversial:** Compliance or audit regimes often require a permanent record of anything that went out the door, even unsigned drafts.
- **Alternative view:** Retain every regenerated version with timestamps and generating user; surface a version list on the application page.
- **Seeking input on:** Whether the current platform's audit posture is comfortable with "only the latest" for pre-signature agreements.

### Synchronous in-request generation, no background queue

- **Decision:** The PDF is produced inside the admin's request (target: under 3 s at p95 for up to 20 items).
- **Why this might be controversial:** Synchronous rendering can tie up app threads and has no retry story if the renderer briefly misbehaves.
- **Alternative view:** Queue the generation; show a "generating" state; return the file when ready.
- **Seeking input on:** Whether expected volume justifies a queue now, or whether the simplicity of synchronous generation is the right call until traffic data says otherwise.

### Hardcoded terms & conditions in the template

- **Decision:** T&C copy lives in the Razor template in the code repository.
- **Why this might be controversial:** Legal-ish wording is typically owned by legal/business, not developers, and may need to change without a deploy.
- **Alternative view:** Store T&C text in the database as a configurable block (the "hybrid" template option that was rejected during brainstorming).
- **Seeking input on:** Whether the business is comfortable with deploy-bound T&C changes, or whether a configurable text block should be in scope.

### Single configured Latin-American locale

- **Decision:** One locale per deployment, sourced from configuration, defaulting to a LatAm style (comma decimal, period thousands).
- **Why this might be controversial:** Applicants could operate across borders; a single locale could render oddly in some contexts.
- **Alternative view:** Per-applicant or per-application locale derived from profile/configuration.
- **Seeking input on:** Is the deployment strictly single-country, or should the locale be selectable?

## Open Questions

- [ ] **Funder identity shape (OQ-002).** Treated as a single configuration block in this spec. If multi-funder scenarios surface, a `Funder` aggregate will likely be warranted.
- [ ] **Reviewer regeneration rights (OQ-004).** Recorded as granted per user direction during brainstorming; re-validate during planning once the full role-scope table for this feature is visible.
- [ ] **Specific default locale code.** Spec commits to a Latin-American *style*; planning should pin a specific code (e.g., `es-CO`, `es-MX`) for E2E-test reproducibility.
- [ ] **T&C copy delivery.** Spec requires hardcoded T&Cs but is silent on who authors the text and when. Planning needs a concrete delivery path.

## Risk Areas

| Risk | Impact | Mitigation |
|------|--------|------------|
| Syncfusion license acquisition delay or commercial cost surprise | Medium | License is a prerequisite for runtime; planning task to confirm acquisition path and configure fail-fast startup check per FR-022 context. |
| Syncfusion HTML-to-PDF has known CSS/font quirks that could cause layout regressions | Medium | Template kept conservative; SC-003 mandates cross-viewer verification in E2E tests. |
| Profile mutation between acceptance and generation produces a "content-stale" agreement | Low | Edge case is explicit; regeneration refreshes. Spec does not commit to capturing an at-acceptance snapshot. |
| Appeal opened after generation leaves a stale agreement accessible | Low | Explicit in Edge Cases and FR-015; prior PDF is retained for audit until regeneration is re-enabled. |
| Concurrent admin + reviewer regeneration on the same application | Low | Optimistic concurrency serializes; losing request sees a retriable error (FR-022). |
| Hardcoded T&C copy may fall out of sync with the legal department | Medium | Flagged as an open question; planning should assign ownership and a delivery checkpoint. |
